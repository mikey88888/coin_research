# AGENTS.md - coin_research

这个仓库允许自动迭代，但必须收敛。

## 工作边界

- 只在本仓库内工作：`/home/thinkpad/quant/coin_research`
- 不要修改仓库外文件
- 不要改 OpenClaw 全局配置
- 不要执行破坏性命令
- 不要自动 push 远端，除非当前明确要求

## 双轨定时规则

这个仓库有两条自动化轨道：

### A. 微改进轨道（高频）

- 节奏：每 30 分钟一次
- 目标：只做 **1 个非常小**、但明确有价值的工程改进
- 优先级：
  1. 提升测试/可验证性
  2. 提升数据同步可靠性
  3. 提升错误提示、参数校验、日志质量
  4. 提升网页可用性
  5. 提升文档清晰度

典型允许动作：

- 增补测试
- 修 bug
- 小范围重构
- 完善 README / 使用说明
- 加轻量 smoke 脚本
- 改善错误提示、参数校验、日志

### B. 策略/因子实验轨道（低频）

- 节奏：每 2 小时一次
- 目标：挑 **1 个最有前途** 的策略或因子实验方向推进
- 输出必须能与历史研究结果做对比
- 活跃榜单只保留最有效的前 10 个结果

历史排名基准文件：

- `research/leaderboard.json`

原始历史实验语料：

- `reports/backtests/`

## 微改进轨道规则

1. 先读仓库现状，提出 **1 个最小但有价值的改进点**
2. 改动前说明目标
3. 若任务属于重构、清理、复杂度压缩、命名优化、代码质量审查，优先套用 `code-quality-guardrails` skill 的原则
4. 只做与该目标直接相关的最小改动
5. 改完后运行：

```bash
uv run pytest -q
```

6. 若测试失败：停止继续扩写，先修当前问题或汇报阻塞
7. 若测试通过：总结改了什么、为什么值得

## 策略/因子实验轨道规则

1. 每轮只推进 **1 个实验主题**
2. 优先顺序：
   - 新因子候选
   - 现有策略参数扫描
   - 新的出场/过滤条件
   - 不同 timeframe / universe 下的稳健性验证
3. 若当前仓库还没有完整实验脚手架，允许先补最小实验基础设施，但不要顺手做大重构
4. 每轮实验结束后，必须把结果整理成可比较结构，并与 `research/leaderboard.json` 当前榜单比较
5. 只有进入前 10 的结果，才保留在活跃榜单中
6. 未进入前 10 的结果，可以留在 `reports/backtests/` 作为历史语料，但不要污染活跃榜单

## 榜单更新原则

更新 `research/leaderboard.json` 时：

- 优先保留 `closed_trades >= 20` 的稳定结果
- 默认比较指标：`annualized_return_pct / max_drawdown_pct`
- 同稳定性分组内，优先保留：
  1. 更高的 return/drawdown 比
  2. 更高的 annualized return
  3. 更低的 max drawdown
- 样本极少的结果只能标记为 `exploratory`，默认排在稳定结果之后
- 活跃榜单最多保留 10 条

## 变更规模控制

- 单轮尽量只动少量文件
- 除非必要，不同时改数据层、回测层、网页层三块
- 优先做可验证、可回滚的小修
- 不要把“微改进轨道”偷偷做成中型重构
- 不要把“实验轨道”偷偷做成全仓架构改造

## 输出要求

每轮结束时给出：

- 本轮优化点 / 实验点
- 改动文件
- 测试结果
- 与历史结果的关系（若是实验轨道）
- 风险 / 后续建议

风格要求：冷静、干练、少废话。
