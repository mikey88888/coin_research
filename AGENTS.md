# AGENTS.md - coin_research

这个仓库允许自动迭代，但必须收敛。

## 工作边界

- 允许在测试通过且完成本地 commit 后自动 push 远端默认分支；若 push 失败，需在日志中写明失败原因

## 总体研究方向

这个项目的目标不是继续围绕单一“五浪加速下跌反转”打转。
目标是做 **更广泛的量化投资研究**，优先建立一批可比较、可迭代的策略/因子候选。

默认原则：

- **五浪策略只是 baseline，不是默认主线**
- 2 小时实验轨道优先扩展新的策略/因子候选
- 只有当候选池足够丰富后，排行榜才真正有意义

## 双轨定时规则

这个仓库有两条自动化轨道：

### A. 微改进轨道（高频）

- 节奏：每 30 分钟一次
- 目标：只做 **1 个**、但明确有价值的工程改进

典型允许动作：

- 增补测试
- 修 bug
- 小范围重构
- 完善 README / 使用说明
- 加轻量 smoke 脚本
- 改善错误提示、参数校验、日志

### B. 策略/因子探索轨道（低频）

- 节奏：每 2 小时一次
- 目标：优先推进 **新的策略/因子探索**，而不是默认继续改五浪
- 输出必须逐步沉淀为：
  - 候选池
  - 可运行实验
  - 可比较结果
- 活跃榜单只保留最有效的前 10 个**已验证结果**

历史排名基准文件：

- `research/leaderboard.json`

探索候选池文件：

- `research/strategy_backlog.md`

原始历史实验语料：

- `reports/backtests/`

## 运行透明度与日志

为了避免“任务执行了，但人看不到过程”，两条轨道都必须写本地日志。

日志目录：

- `log/`

对应文件：

- 微改进轨道 → `log/micro_optimization.log`
- 实验轨道 → `log/experiment_ranking.log`

日志规则：

- **只追加，不覆盖**
- **优先写人类可读文本**，不要优先写 JSON
- 每轮开始时就先写一条“本轮准备做什么”
- 每轮结束时再补“实际上做了什么、结果如何”
- 如果因为已有活动任务而跳过，也要写一条 skip 记录
- 如果阻塞，也要写清阻塞点

每轮日志至少包含：

- 时间
- 轨道类型
- 本轮目标
- 选择原因
- 实际运行的关键命令
- 改动文件
- 测试 / 回测结果
- 是否更新 `research/leaderboard.json`
- 是否 skip / blocked / done

## 微改进轨道规则

1. 先读仓库现状，提出 **1 个最小但有价值的改进点**
2. 在 `log/micro_optimization.log` 先追加一条 start 记录
3. 若任务属于重构、清理、复杂度压缩、命名优化、代码质量审查，优先套用 `code-quality-guardrails` skill 的原则
4. 只做与该目标直接相关的最小改动
5. 改完后运行：

```bash
uv run pytest -q
```

6. 若测试失败：停止继续扩写，先修当前问题或汇报阻塞，并写入日志
7. 若测试通过：总结改了什么、为什么值得，并写入日志

## 策略/因子探索轨道规则

1. 每轮只推进 **1 个探索主题**
2. 在 `log/experiment_ranking.log` 先追加一条 start 记录
3. **默认不要继续围绕五浪 baseline 做小修小补**，除非本轮明确是在拿它做对照
4. 如果当前候选池不够丰富，优先做以下事情，而不是先刷排行榜：
   - 新策略想法归档
   - 新因子假设归档
   - 最小实验脚手架
   - 新 baseline / 新过滤条件 / 新出场逻辑的 prototype
5. 优先顺序：
   - OHLCV 可直接支持的新策略候选
   - 横截面因子候选
   - 趋势 / 反转 / 突破 / 波动率 / 成交量类因子
   - regime filter（市场状态过滤）
   - 现有策略的参数扫描与稳健性验证
6. 若当前仓库还没有完整实验脚手架，允许先补最小实验基础设施，但不要顺手做大重构
7. 每轮实验结束后，必须做下面三件事中的至少一件：
   - 更新 `research/strategy_backlog.md`
   - 产出一个新的最小实验实现
   - 产出一个可比较的实验结果
8. 只有进入前 10 的结果，才保留在 `research/leaderboard.json` 活跃榜单中
9. 未进入前 10 的结果，可以留在 `reports/backtests/` 作为历史语料，但不要污染活跃榜单
10. 无论实验是否成功，都要把结果或阻塞写入日志

## 候选池优先原则

当候选策略数量不足时，优先扩池，不急着排名。

建议默认先探索这些方向：

- Donchian / breakout
- 均线趋势跟随
- 多周期趋势共振
- z-score 均值回归
- 波动率压缩后突破
- 量价异常后的延续 / 反转
- ATR 风险控制框架
- 横截面相对强弱
- regime filter（趋势市 / 震荡市切换）
- 基于 OHLCV 的动量 / 反转复合因子

## 榜单更新原则

更新 `research/leaderboard.json` 时：

- 它表示的是 **已验证的活跃 top 10**，不是全部想法池
- 优先保留 `closed_trades >= 20` 的稳定结果
- 默认比较指标：`annualized_return_pct / max_drawdown_pct`
- 同稳定性分组内，优先保留：
  1. 更高的 return/drawdown 比
  2. 更高的 annualized return
  3. 更低的 max drawdown
- 样本极少的结果只能标记为 `exploratory`，默认排在稳定结果之后
- 活跃榜单最多保留 10 条

## 变更规模控制

- 单轮尽量只动少量文件
- 除非必要，不同时改数据层、回测层、网页层三块
- 优先做可验证、可回滚的小修
- 不要把“微改进轨道”偷偷做成中型重构
- 不要把“实验轨道”偷偷做成全仓架构改造

## 输出要求

每轮结束时给出：

- 本轮优化点 / 探索点
- 改动文件
- 测试结果
- 与历史结果的关系（若是实验轨道）
- 是否新增候选 / 是否更新榜单
- 风险 / 后续建议

风格要求：冷静、干练、少废话。
